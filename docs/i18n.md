# Internacionalización e idioma base

> COE está pensado para uso **global**, no atado a un idioma concreto. Este documento fija principios transversales. La etapa **L0** (normalización de idioma en Ingest) se detalla en [l0-ingest.md](l0-ingest.md).

**Estado:** ✅ Aprobado · spec operativa en [l0-ingest.md](l0-ingest.md) ✅

## Alineación con la visión fundacional

| Principio ([COE].md) | Cómo lo cumple i18n |
|------------------------|---------------------|
| Optimizar **contexto** a escala global | Núcleo agnostic; despliegue configurable (`target_lang`, locale packs) |
| **Cambiar representación** en N1–N5, no traducir al final | L0 **pre-N1** sobre prosa; Model Adapter no sustituye idioma |
| CIR / representación intermedia | N1 Unicode-neutral; N2–N3 vía locale; N4–N5 labels post-L0 |
| El LLM es el consumidor | Contexto hacia el modelo en `target_lang`; benchmarks A/B en ese idioma |
| No atarse a un idioma de origen del equipo | `en` como referencia del repo; `zh` y otros vía packs — no español por defecto |

[COE]: Context%20Optimization%20Engine%20(COE).md

## Principios

1. **Núcleo language-agnostic** — algoritmos, contratos de datos y pipeline N1–N5 independientes del idioma.
2. **Locale packs** — patrones léxicos (factorización N2, relaciones N3, segmentación) en plugins configurables, no hardcodeados.
3. **Idioma de referencia del repo** — specs, ejemplos canónicos y tests de referencia en **inglés** (`en`). Otros idiomas vía locale packs y datasets.
4. **Traducción temprana, no tardía** — si el despliegue usa un idioma base distinto al de entrada, traducir **antes** de N1, sobre prosa natural; no traducir representaciones ya comprimidas (post-N2/N3).
5. **Tests de comprensión en `target_lang`** — benchmarks A/B (desde N2) con el mismo idioma que verá el LLM destino.
6. **Tres ejes de idioma separados** — contexto hacia el LLM, idioma del usuario y idioma de respuesta no son lo mismo (ver § siguiente).

## Tres ejes de idioma (no confundir)

COE y PCM operan con **tres decisiones distintas**. Mezclarlas es la causa habitual de respuestas en idioma incorrecto o de benchmarks mal interpretados.

| Eje | Parámetro | Quién lo fija | Qué controla |
|-----|-----------|---------------|--------------|
| **Contexto → LLM** | `target_lang` (+ L0) | Despliegue / Gateway | Idioma del **contexto optimizado** que lee el modelo |
| **Usuario** | `user_lang` | Detección por turno (mensaje del usuario) | Idioma en que **habla** la persona en este turno |
| **Respuesta → usuario** | `response_lang` | PCM / system prompt (Gateway puede inyectarlo) | Idioma en que el LLM **debe redactar la respuesta** |

### ¿Respondemos en el idioma del contexto original?

**No por defecto.** La respuesta debe ir en el idioma en que **el usuario se dirige a nosotros en este turno** (`user_lang` / `response_lang`), no en el idioma del bundle de contexto ni en `target_lang`.

| Situación | Contexto hacia LLM (`target_lang`) | Respuesta al usuario (`response_lang`) |
|-----------|-------------------------------------|----------------------------------------|
| Usuario escribe en español; RAG en inglés; L0 → `en` | Inglés (normalizado) | **Español** (idioma del mensaje del usuario) |
| Usuario escribe en inglés; contexto ya en inglés | Inglés | Inglés |
| Usuario mezcla ES/EN en el mismo mensaje | Según L0 (unificado o parcial) | **Idioma dominante del mensaje** o override explícito |
| Glosario bilingüe intencional en contexto | Bloques preservados (ver L0) | Sigue **`user_lang` del turno**, no el del glosario |

**PCM / system** debe incluir siempre una instrucción explícita del tipo: «Responde al usuario en `{response_lang}`», derivada del mensaje actual (o de preferencia de perfil). COE **no** sustituye esa instrucción: solo optimiza el bloque de contexto.

```python
# Gateway (orientativo) — responsabilidades separadas
optimize_context(blocks=[...], target_lang="en", locale="en", levels=[1, 2])
# → contexto optimizado en inglés

# PCM / system (fuera de COE)
# response_lang = detect(user_message)  # p. ej. "es"
# "Answer the user in Spanish. Context below may be in English."
```

El KPI `user_language_match` en [benchmarks.md](benchmarks.md) valida precisamente esto: la respuesta B debe estar en **`response_lang` del caso**, no en el idioma del contexto original.

## Contexto multilingüe mezclado

Cuando el usuario (o las fuentes RAG/herramientas) aportan **partes en distintos idiomas** en el mismo bundle, la defensa es **por capas**:

### 1. L0 activo (política por defecto en producción)

| Paso | Comportamiento |
|------|----------------|
| Detección | **Por bloque** (`detected_lang` en metadatos) |
| Traducción | Cada bloque con idioma ≠ `target_lang` y confianza ≥ umbral → traducir a `target_lang` |
| Skip | Bloque ya en `target_lang`; bloques con `preserve_lang: true` (glosario, citas legales) |
| Baja confianza | **No traducir** el bloque; `ingest_trace.warning`; puede quedar mezcla residual |
| Salida | Un solo idioma de contexto hacia N1–N5 salvo bloques marcados para preservar |

Así el pipeline N2+ recibe un locale pack coherente. La mezcla del usuario **en el contexto de entrada** se resuelve antes de comprimir.

### 2. Bundles multilingües intencionados

Bloques con metadato `preserve_lang: true` o `source_type: glossary` **no se traducen** en L0. El Renderer puede etiquetarlos («Original: … / Translation: …») si el despliegue lo requiere. El resto del bundle sigue unificándose a `target_lang`.

### 3. L0 desactivado o mezcla residual

Si el bundle llega mezclado a N1:

- **`locale`** lo fija el cliente o el **idioma dominante** del bundle (por tokens o por número de bloques).
- `ingest_trace` registra `mixed_bundle: true` y idiomas detectados.
- N2/N3 usan **un** locale pack; el otro idioma se procesa peor (passthrough parcial, menos factorización) — aceptable como degradación documentada, no como fallo silencioso.

### 4. Usuario que mezcla idiomas **entre turnos** (diálogo)

Distinto de contexto mezclado: el usuario escribe un turno en español y otro en inglés.

| Regla | Acción |
|-------|--------|
| **`response_lang` por turno** | Redetectar en **cada mensaje del usuario**; no heredar el idioma del contexto acumulado en N5 |
| **State Store (N5)** | Persiste hechos en grafo largely agnostic; `render_prose()` en `target_lang` del despliegue |
| **Instrucción PCM** | Actualizar `response_lang` cada turno; el LLM puede leer contexto en EN y responder en ES |

El store no impone idioma de respuesta: impone coherencia del **contexto interno**. La presentación al usuario la manda PCM.

### 5. Resumen de defensa

```
Entrada mezclada (contexto)
    → L0 por bloque → target_lang unificado (+ preserve_lang donde toque)
    → locale pack alineado con target_lang
    → benchmarks con casos multilingual[] en benchmarks.md

Usuario mezcla idiomas (diálogo)
    → response_lang = idioma del mensaje actual (PCM)
    → contexto puede estar en target_lang ≠ response_lang (explícito en system)
    → user_language_match en benchmark E2E
```

## ¿Por qué traducir antes de comprimir?

| Momento | Forma del texto | Calidad de traducción típica |
|---------|-----------------|------------------------------|
| **Pre-N1 (L0)** | Oraciones naturales completas | Alta — MT/LLM entrenados en prosa |
| **Post-N1** | Líneas compactas, `clave=valor` | Media |
| **Post-N2+** | Árboles, `entity:`, grafos | Baja — formato no estándar |

El lector final es un LLM; muchos despliegues usan **inglés** o **chino** como idioma efectivo del contexto aunque la fuente sea otra. Normalizar idioma al inicio evita obligar al traductor a interpretar nuestras representaciones internas.

## `target_lang` (configuración de despliegue)

Parámetro de **Ingest / L0**, no de un nivel de optimización:

| Valor | Uso típico |
|-------|------------|
| `en` | Default global; mayoría de LLM y documentación del proyecto |
| `zh` | Despliegues orientados a chino; requiere locale pack y segmentación propios |
| `auto` | Detectar idioma dominante del bundle; traducir solo si ≠ `target_lang` explícito del cliente |
| (omitir L0) | Entrada ya en idioma correcto; pipeline N1→N5 directo |

No hay un único idioma “correcto” para todo el mundo: **`target_lang` es decisión de despliegue**, no constante del código.

## Locale packs

Contenido previsto por locale (directorio futuro `src/coe/locales/` o similar):

```
locales/
├── en/
│   ├── factorization_patterns.yaml   # N2: "X works at Y", …
│   ├── relation_patterns.yaml        # N3
│   └── segmentation.yaml             # líneas / oraciones
├── zh/
│   ├── factorization_patterns.yaml
│   └── segmentation.yaml             # sin espacios; reglas distintas a N1 por línea
└── es/                               # opcional; no default del repo
    └── …
```

El **núcleo** no importa un locale por defecto en runtime: el Gateway recibe `locale` + `target_lang` en `OptimizeOptions`.

## Implicaciones por nivel

| Etapa | Dependencia de idioma |
|-------|------------------------|
| **L0** | Detección + traducción opcional → `target_lang` |
| **N1** | Sintáctico (líneas); funciona en cualquier script Unicode; normalización Unicode-neutral |
| **N2** | **Locale pack** — patrones sujeto-verbo/atributo |
| **N3** | **Locale pack** — patrones de relación |
| **N4–N5** | Modelo de grafo largely agnostic; labels pueden conservar idioma post-L0 |
| **Renderer** | Serializa en idioma del bundle (ya normalizado por L0) |

## Chino (`zh`) — notas específicas

No es solo “traducir al chino”:

- Segmentación sin espacios entre palabras.
- N1 por **línea** puede ser insuficiente; puede requerir segmentador de oraciones en Normalizer.
- Patrones N2/N3 distintos al inglés; locale pack obligatorio.
- Tests de comprensión con LLM evaluador acostumbrado a chino (Ollama con modelos multilingües, etc.).

## Documentación vs. runtime

| Ámbito | Idioma |
|--------|--------|
| Docs de diseño (pueden estar en español para el equipo) | Cualquiera |
| Specs, ejemplos en `data/`, casos de test de referencia | **Inglés** preferido |
| Contexto optimizado hacia el LLM | **`target_lang` del despliegue** |
