# Internacionalización e idioma base

> COE está pensado para uso **global**, no atado a un idioma concreto. Este documento fija principios transversales. La etapa **L0** (normalización de idioma en Ingest) se detalla en [l0-ingest.md](l0-ingest.md).

## Principios

1. **Núcleo language-agnostic** — algoritmos, contratos de datos y pipeline N1–N5 independientes del idioma.
2. **Locale packs** — patrones léxicos (factorización N2, relaciones N3, segmentación) en plugins configurables, no hardcodeados.
3. **Idioma de referencia del repo** — specs, ejemplos canónicos y tests de referencia en **inglés** (`en`). Otros idiomas vía locale packs y datasets.
4. **Traducción temprana, no tardía** — si el despliegue usa un idioma base distinto al de entrada, traducir **antes** de N1, sobre prosa natural; no traducir representaciones ya comprimidas (post-N2/N3).
5. **Tests de comprensión en `target_lang`** — benchmarks A/B (desde N2) con el mismo idioma que verá el LLM destino.

## ¿Por qué traducir antes de comprimir?

| Momento | Forma del texto | Calidad de traducción típica |
|---------|-----------------|------------------------------|
| **Pre-N1 (L0)** | Oraciones naturales completas | Alta — MT/LLM entrenados en prosa |
| **Post-N1** | Líneas compactas, `clave=valor` | Media |
| **Post-N2+** | Árboles, `entity:`, grafos | Baja — formato no estándar |

El lector final es un LLM; muchos despliegues usan **inglés** o **chino** como idioma efectivo del contexto aunque la fuente sea otra. Normalizar idioma al inicio evita obligar al traductor a interpretar nuestras representaciones internas.

## `target_lang` (configuración de despliegue)

Parámetro de **Ingest / L0**, no de un nivel de optimización:

| Valor | Uso típico |
|-------|------------|
| `en` | Default global; mayoría de LLM y documentación del proyecto |
| `zh` | Despliegues orientados a chino; requiere locale pack y segmentación propios |
| `auto` | Detectar idioma dominante del bundle; traducir solo si ≠ `target_lang` explícito del cliente |
| (omitir L0) | Entrada ya en idioma correcto; pipeline N1→N5 directo |

No hay un único idioma “correcto” para todo el mundo: **`target_lang` es decisión de despliegue**, no constante del código.

## Locale packs

Contenido previsto por locale (directorio futuro `src/coe/locales/` o similar):

```
locales/
├── en/
│   ├── factorization_patterns.yaml   # N2: "X works at Y", …
│   ├── relation_patterns.yaml        # N3
│   └── segmentation.yaml             # líneas / oraciones
├── zh/
│   ├── factorization_patterns.yaml
│   └── segmentation.yaml             # sin espacios; reglas distintas a N1 por línea
└── es/                               # opcional; no default del repo
    └── …
```

El **núcleo** no importa un locale por defecto en runtime: el Gateway recibe `locale` + `target_lang` en `OptimizeOptions`.

## Implicaciones por nivel

| Etapa | Dependencia de idioma |
|-------|------------------------|
| **L0** | Detección + traducción opcional → `target_lang` |
| **N1** | Sintáctico (líneas); funciona en cualquier script Unicode; normalización Unicode-neutral |
| **N2** | **Locale pack** — patrones sujeto-verbo/atributo |
| **N3** | **Locale pack** — patrones de relación |
| **N4–N5** | Modelo de grafo largely agnostic; labels pueden conservar idioma post-L0 |
| **Renderer** | Serializa en idioma del bundle (ya normalizado por L0) |

## Chino (`zh`) — notas específicas

No es solo “traducir al chino”:

- Segmentación sin espacios entre palabras.
- N1 por **línea** puede ser insuficiente; puede requerir segmentador de oraciones en Normalizer.
- Patrones N2/N3 distintos al inglés; locale pack obligatorio.
- Tests de comprensión con LLM evaluador acostumbrado a chino (Ollama con modelos multilingües, etc.).

## Documentación vs. runtime

| Ámbito | Idioma |
|--------|--------|
| Docs de diseño (pueden estar en español para el equipo) | Cualquiera |
| Specs, ejemplos en `data/`, casos de test de referencia | **Inglés** preferido |
| Contexto optimizado hacia el LLM | **`target_lang` del despliegue** |
